1. Introduction

1.1 Purpose

The purpose of this document is to present a detailed description of our software product ‘NSU-CLS’.It will explain the purpose and features of the software, what the software will do and the constraints under which it must operate. This document describes the project's target audience and its user interface, hardware and software requirements.This document is intended for users of the software.

1.2 Document Conventions

This Document was created based on the IEEE template for System Requirement Specification Documents.

1.3 Intended Audience and Reading Suggestions

The project is about a complaint lodging system, and it is restricted within the university premises. This document is intended for the course instructor, and the project is implemented under his guidance. This SRS contains the overall description of the project, such as product scope, product functions, design and implementation. The project can be helpful for the faculty members, students, teaching assistants, admin members, and other North South University staff members.

1.4 Product Scope

The project is about a complaint lodging system, and it is restricted within the university premises. This document is intended for the course instructor, and the project is implemented under his guidance. This SRS contains the overall description of the project, such as product scope, product functions, design and implementation. The project can be helpful for the faculty members, students, teaching assistants, admin members, and other North South University staff members. The purpose of the complaint lodging system is to manage all the complaints of faculty members, students, teaching assistants, admin, and other staff members of North South University to create a user-friendly and convenient application. We hope to provide an easy-to-use application so that users can lodge their complaints without any hassle.

2. Overall Description

2.1 Product Perspective

The product name is NSU-CLS. The project is a complaint lodging system of North South University. This system manages all the complaints lodged by faculty members, students, teaching assistants, admin, and other university staff members. Faculty members and admin can review the complaints. It also contains the user's name, NSU-Id, email address, and additional information. This is a new self-contained product.

2.2 Product Functions

  • Users need to register at first to complain.
  • Users like students, faculty members, teaching assistants, and admin members must register with their NSU email.
  • If the user is already registered, they need to login into the system.
  • Users can lodge their complaints with proper evidence and choose a reviewer for the complaint. Students, faculty members, teaching assistants, admin members, and other staff members of North South University can lodge complaints.
  • Only faculty members and admin can review complaints.
  • A lodger can edit or delete a complaint and change the reviewer for the complaint.
  • The system also stores all the information about the user.

2.3 User Classes and Characteristics

  • Users like students, faculty members, teaching assistants, admin members, and other staff members can lodge complaints and edit or remove them.
  • Only faculty members and admin can review a complaint.
  • Only the sysadmin or the lodger himself can remove a complaint.

2.4 Operating Environment

  • Platform: Web and Android
  • Web Server: Any operation system with NodeJS RTE 16 or newer installed.
  • Client System: Any system with a modern web browser or for android with android version 7.0+
  • Database Configuration: MySQL

2.5 Design and Implementation Constraints

The backend of NSU-CLS is developed using NodeJS (ExpressJS), and the frontend part is built on ReactJS. The backend and frontend are connected through well-written APIs. For mobile, the android application is developed using Java. People using devices with Android versions lower than seven can not use the android version of this application.

2.6 User Documentation

The product will provide users with basic running instructions. There is a part named ‘How to Complain’ where users will get guidelines on how they can lodge a complaint.

2.7 Assumptions and Dependencies

The user must have an active internet connection to utilize the system. For web users, there must be a browser to access this product. Mobile users must have a minimum of Android version 7 installed. A user can lodge a complaint then. The user can lodge the complaint through speech using the android app. After lodging a complaint, the user, the reviewer and sysadmins will be notified via email.

3. External Interface Requirements

3.1 User Interfaces

The system will take the user to the home page first. Then the user needs to sign in to the system with the correct credentials. If the user is not registered, he will be redirected to the sign-up page, and he needs to register to the system with all the correct information. A user can also sign up and sign in through Google sign-up and Google sign-in. Students, faculty members, admin members, teaching assistants have to sign up using NSU email. After signing in successfully a user can lodge complaints, and there is an option to choose a reviewer for the complaints (which is mandatory). In the case of an android application, a user can lodge Complaint through speech as there is a Bangla speech recognizer that converts Speech to text. A user can lodge complaints against multiple people. An email will be sent to the lodger, reviewer, and sysadmin to notify them. The lodger should also attach proper evidence. Moreover, the lodger has the opportunity to edit, remove or change the reviewer. The user's information will be stored, including their image(which will be taken from their NSU-ID's scan copy) in the user's profile. There will be a Logout button to exit from the system.

3.2 Hardware Interfaces

The mobile app version of the software might need a microphone to convert speech to text.

3.3 Software Interfaces

‘NSU-CLS’ mainly relies on the following:

  • ExpressJS - Express is a minimal and flexible Node.js web application framework that provides a robust set of features for web and mobile applications.It will be used to build the backend of our application.
  • ReactJS - React is a free and open-source front-end JavaScript library for building user interfaces based on UI components. It will be used to build the UI or frontend of the application.
  • MySQL - This is the DBMS we will use to store our data.
  • Android - Our mobile app will be built using Android.

3.4 Communications Interfaces

The application’s user interface will send the data to the backend using rest API. The android app will also use speech to text API to get data through speech. The whole communication will be done via HTTP. Our authentication system will be used to log in or register.

4.1.1 Sign up

Stimulus/ Response Sequences

The following expanded use case shows the response sequences between the actor actions and the system responses.
  • Use Case: Signup
  • Preconditions: The user must be someone from North South University
  • Actor(s): User, Reviewer
  • Goal: To create an account to the system.

Overview: To use the system the user needs to sign up. The user can easily sign up by filling the registration form

Cross Reference: Not applicable.

Typical Course of Events:

Actor's Action System's Response
1. User visits the signup page 2. System responses with the sign up page
3. The user will create an account by filling out the registration form 4. System will verify the given information and save the details to the database. Then the system will send verification email to that email address
5. After verifying the verification email, the user successfully signed up and ready to use the system

Alternate course of action:

Step 5: The credentials provided by the user may not be correct or already in the database, in that case the system will show an error page.

4.3 Lodge complaints

Stimulus/ Response Sequences

The following expanded use case shows the response sequences between the actor actions and the system responses.
  • Use Case: Lodge a complaint
  • Preconditions: The user must login to the system using correct credentials in order to lodge a complaint.
  • Actor(s): User
  • Goal: To lodge a complaint against someone with proper evidence.

Overview: A user can edit or remove any complaint which was lodged by him. Moreover, a lodger can choose a reviewer for the complaints.

Cross Reference: Not applicable.

Typical Course of Events:

Actor's Action System's Response
1. After successful login, user requests for the lodge a complaint page. 2. The system will generate the lodge complaint page
3. The user searches for the person against whom the complaint will be lodged. Users can lodge complaints against multiple users. 4. The system will find users based on the searched keyword.
5. User searches for the person whom he/she wants to choose as reviewer. 6. The system will find the reviewer's name based on the searched keyword.
7. User will lodge complaint and provide proper evidences. 8. Then the system will store the lodged complaint and the evidences in database. The system will also notify SysAdmins, Reviewer and User via email

Alternate course of action:

Step 2 ,3 ,4 and 5: If the keyword does not match i.e valid names are not found after searching ,then a complaint cannot be lodged.

Step 7 and 8:The message and evidence part cannot be kept blank.

  • Use Case: Edit a complaint
  • Preconditions: The user must login to the system using correct credentials in order to edit a complaint. Also the user must have lodged a complaint before.

  • Goal: To edit a complaint that has been lodged earlier

Overview: A user can edit or remove any complaint which was lodged by him. Moreover, a lodger can choose a reviewer for the complaints.

Actor's Action System's Response

3. The user then selects a specific complaint and clicks on edit button

5. Other Nonfunctional Requirements

5.1 Performance Requirements

The system must be capable of running the application fluently. There should be no visible latency between the user's action and the system's response. In case of mobile app, it needs to be lightweight and responsive

5.2 Safety Requirements

To ensure that no one of NSU-CLS’s users loses any data while using NSU-CLS (due to a crash or a bug of some kind) the developer team updates NSU-CLS regularly.

5.3 Security Requirements

The system shall not disclose any personal information about the users. The Application shall not grant access to an unauthorized user or any user outside North South University.

5.4 Software Quality Attributes

  • AVAILABILITY: Since it will be an online system it needs to be regularly available to the user.
  • USABILITY: NSU-CLS provides the users with both simple and advanced features. The user should have no trouble navigating and using the system.
  • RELIABILITY: The user can rely on the NSU-CLS as the user can lodge complaints without letting the person knowing against whom complaint has been lodged.

5.5 Business Rules

The website will be free for all. Every user will get all features for free.
